AI Incident Copilot

An AI-powered observability and incident-response platform for microservices. It simulates a production-style distributed system, collects logs, metrics, and traces, detects anomalies from Prometheus data, and generates incident reports with root-cause hypotheses and recommended actions — all surfaced in a Streamlit copilot dashboard.

View on GitHub ↗
Problem

Incident response is a lot of manual checking.

In real engineering teams, responding to an incident usually means manually checking logs, metrics, dashboards, traces, recent deployments, service dependencies, error rates, and latency spikes — one tab at a time. AI Incident Copilot explores how much of that first pass an assistant can do automatically: collect the signals, detect the anomaly, summarize the likely cause, recommend next steps, and save a snapshot for later review.

Architecture

The detection pipeline

Traffic hits the FastAPI services, which expose Prometheus metrics. The anomaly detector queries those metrics on demand, and the Streamlit dashboard calls the detector to render (and optionally save) an incident report.

What This Proves

Core capabilities under test

This isn't just a "is it running" check — it's a walkthrough of the full incident detection pipeline, end to end.

→Services stay online even while showing elevated error rates or latency — the failure is in behavior, not availability.
→Incidents get detected from metrics — the anomaly detector reasons over Prometheus data, not manual alerts.
→The affected service is identified by name, not just "something is wrong."
→Traces help localize the problem — Jaeger shows which requests were slow or failed, down to the span.
→A summary and next steps get generated automatically — root cause hypothesis plus a ranked action list.
Test Walkthrough

Running the pipeline end to end

Starting from a healthy baseline, injecting a real failure, and following the signal all the way to an AI-generated incident report.

1
Baseline — confirm everything is healthy
Prometheus is actively scraping all three services, and a basic up query confirms each target is reporting in.
Status → Target health
Query: up
2
Inject a real failure
Each service ships a /simulate/error endpoint. Hitting it on order-service via its Swagger UI returns a real 500 — the failure this whole pipeline is meant to catch.
3
Metrics pick up the anomaly
order-service's request and error counters are visible in Prometheus, and the request-rate graph shows a clear spike right when the failure traffic hit.
Query: {__name__=~".*request.*"}
sum by (service) (rate(http_requests_total[1m]))
4
Grafana visualizes it across services
The AI Incident Copilot Grafana dashboard rolls up total requests, request rate, average latency, and error count — order-service's error count panel spikes while the others stay flat.
5
Anomaly detector generates an incident report
Hitting the detector's /report endpoint directly returns a structured incident: which service, what type of anomaly, severity, the actual error count vs. threshold, and a plain-language message.
6
The copilot dashboard surfaces it
The Streamlit dashboard is the single pane of glass — System Status, Active Incidents, Summary, Root Cause Hypothesis, and an LLM-generated analysis all populate from the same detector output shown above. (Screenshot below shows the healthy baseline view — this is the UI shell that renders the incident content once one is active.)
7
Jaeger traces pinpoint the request path
Filtering Jaeger to order-service surfaces individual traces with span counts and durations — the tool to drill from "order-service is failing" down to the exact request.
