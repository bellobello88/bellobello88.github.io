Botrock — AWS Generative AI Experiments

Three small, serverless experiments exploring what Amazon Bedrock can do out of the box — text generation, text-to-image, and retrieval-augmented Q&A — each built as a standalone AWS Lambda function.

View on GitHub ↗
Problem

Getting hands-on with Bedrock's model zoo.

Amazon Bedrock gives access to several foundation models — text, image, and retrieval-based — behind one API, but each has a different request shape, response format, and failure mode. Rather than reading about it, this repo is three small Lambda functions built to actually call each model type, handle its response, and wire it into a usable output: generated text, a stored image, or a grounded answer.

Architecture

The shared pattern

All three functions follow the same shape: a Lambda receives a prompt, calls a Bedrock model, and shapes the response for the caller — with S3 and a Knowledge Base layered in where the use case needs storage or retrieval.

Three Experiments

What each function does

1

Text Generation

Cohere Command Light

Takes a prompt, calls Bedrock's Cohere model, and returns generated text. The simplest of the three — built first to get the invoke/response round-trip working before adding storage or retrieval.

boto3 bedrock-runtime invoke_model
# request shape client.invoke_model( modelId='cohere.command-light-text-v14', body=json.dumps({ "prompt": input_prompt, "temperature": 0.9, "maxTokens": 1000 }) )
2

Movie Poster Generator

Stable Diffusion XL

Text-to-image: a prompt goes in, Bedrock's Stable Diffusion XL model generates a poster, and the Lambda decodes the base64 image, uploads it to S3, and returns a presigned URL so the caller can view it without the bucket being public.

Stable Diffusion XL S3 Presigned URLs
# decode + store image_bytes = base64.b64decode(artifact['base64']) s3.put_object(Bucket='1movieposterdesign1', Key=poster_name, Body=image_bytes) s3.generate_presigned_url('get_object', ExpiresIn=3600)
3

Knowledge Base Q&A

Claude Instant v1 · RAG

The most involved of the three: documents in S3 feed a Bedrock Knowledge Base, which chunks and embeds them into a vector store. A Lambda calls retrieve_and_generate, so answers come back grounded in the source documents instead of the model's own memory — a small RAG pipeline behind an API Gateway endpoint.

Knowledge Base RAG API Gateway Vector store
# grounded Q&A bedrock_agent.retrieve_and_generate( input={'text': user_prompt}, retrieveAndGenerateConfiguration={ 'type': 'KNOWLEDGE_BASE', 'knowledgeBaseConfiguration': { 'knowledgeBaseId': knowledge_base_id, 'modelArn': claude_instant_arn } } )
Technical Decisions

Why it's built this way

Serverless by default

  • Each experiment is a standalone Lambda — no servers to manage for infrequent, bursty AI calls
  • IAM roles scoped per function to the minimum Bedrock, S3, and CloudWatch permissions needed
  • API Gateway in front of the Q&A function so it's callable over HTTP without exposing AWS credentials

Model choice

  • Cohere Command Light for fast, cheap text generation while iterating on the Lambda plumbing
  • Stable Diffusion XL for image quality on a fixed prompt style (movie posters)
  • Claude Instant v1 for the RAG answer step — fast enough for a Q&A endpoint, good at grounded synthesis
Challenges & Trade-offs

What was hard

Every Bedrock model has its own request and response shape — Cohere's payload isn't the same as Stable Diffusion's, and the Knowledge Base API is different again. Most of the actual debugging time went into reading each model's response structure correctly (base64 image payloads especially) rather than the AWS plumbing around it.

For the poster generator, returning images directly through Lambda wasn't practical, so the trade-off was routing through S3 with a presigned URL — one extra hop, but it keeps the bucket private and the response payload small.
