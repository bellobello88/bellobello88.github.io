Customer Churn MLOps Pipeline

An MLOps pipeline for customer churn prediction — PySpark for data processing, a PyTorch MLP for classification, and MLflow tracking every run's parameters, metrics, and model versions end to end.

View on GitHub ↗
Problem

Getting past "it works on my validation set."

A churn model is easy to fool with itself — post a great validation score and it's tempting to call it done, even if what it learned doesn't hold up on anything outside that slice of data. This pipeline treats that risk as something to design against from the start, not catch later. Training and internal validation stay in one split; a separate external test set exists purely to answer one question — did the model generalize, or did it just memorize the validation set?

Architecture

Data → training → tracking

PySpark handles the data-processing stage on the raw Kaggle dataset, a PyTorch MLP is trained on the result, and every run — parameters, metrics, and the resulting model — is logged to MLflow for comparison and versioning.

Tracking a model from run to comparison

MLflow is the through-line here — every experiment, hyperparameter, and metric shown below is logged automatically as part of the training run.

1
MLflow home — experiment tracking set up
The "Customer Churn Prediction" experiment, with tracing, evaluation, and prompt tooling available alongside standard model-training tracking.
2
A single run, fully logged
21 metrics per run — validation and internal-test accuracy, precision, recall, F1, ROC AUC — plus tags recording the framework (PyTorch), data processing engine (PySpark), task type, and a flagged distribution_shift_detected: true.
3
Hyperparameters, version-controlled
Batch size, epochs, learning rate, and the train/validation/internal-test split ratio are all defined in source, not tweaked ad hoc.
4
Multiple runs, each versioned
Three runs sweeping the learning rate, each registering a new version of CustomerChurnModel in the registry — v1, v4, v5 — so any run's model is retrievable later. Each sweep is just two lines changed before re-running the script: the learning rate, and a matching run name so it's identifiable in MLflow later.
# swept across three runs LEARNING_RATE = 0.001 # → RUN_NAME = "baseline-mlp-8epochs" LEARNING_RATE = 0.002 # → RUN_NAME = "mlp-lr-002-v2" LEARNING_RATE = 0.005 # → RUN_NAME = "mlp-lr-0005-v2"
5
Comparing runs visually
A parallel coordinates plot lines up learning rate against validation loss across the three runs — useful for spotting whether a hyperparameter sweep is trending the right direction before diving into tables.
6
Run comparison — where it gets honest
Side-by-side external test metrics across the three runs. This is the view that surfaced the distribution-shift flag — see below.
Challenges & Trade-offs

What the external test set caught

Internal validation numbers look strong across every run — accuracy above 98%, ROC AUC above 0.99. Run the same models against the external test split, though, and accuracy drops to roughly 50–51% — essentially chance for a binary classifier. That gap is exactly what the distribution_shift_detected tag is flagging: the model learned patterns specific to the internal split that don't transfer.

Open problem

Tuning the learning rate alone didn't close the gap — external accuracy stayed pinned near chance across all three runs. That points to the issue being about data (feature distribution, leakage, or split construction) rather than model capacity or optimization, and it's the natural next thing to dig into.
